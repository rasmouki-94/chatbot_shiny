========================
A) BARRE DE PROGRESSION — DOIT MARCHER (dynamique + %)
========================
Symptôme : la barre n’avance pas ou se bloque.

Exigences UX
- Style Apple ultra épuré, fine, arrondie, accent #8e66c7.
- Afficher un pourcentage discret (ex: “42%”) dans le header, aligné à droite.
- Animation douce (transition 250–400ms), sans clignotement.
- Fonctionne sans dépendre d’un re-render complet Shiny.

Exigences techniques (non négociables)
- Implémenter la progression via un canal front stable :
  1) Le server calcule `pct` (0–100) à chaque étape et envoie uniquement un custom message.
     session$sendCustomMessage("progress_update", list(pct = pct))
  2) Le front (JS) écoute et met à jour DOM :
     - width du fill
     - texte du pourcentage
- Important : la barre et son fill DOIVENT avoir des IDs DOM stables, jamais recréés par `renderUI` à chaque message.
- Si actuellement la barre est dans un `uiOutput` qui rerender => la déplacer dans un bloc UI statique (tags$div fixe) et ne mettre à jour que via JS.

Calcul de progression (source unique de vérité)
- Questions : 0% → 75% linéaire sur N questions réelles (N = nb questions effectif)
  pct_questions = (index_question_terminee / N) * 75
- Lead gate (prénom/nom/entreprise/email) : 75% → 90% linéaire sur 4 étapes
- Résultat affiché : 100%
- À chaque réponse validée, appeler une fonction server `push_progress(step)`.

À faire
1) Diagnostiquer pourquoi ça ne bouge pas (IDs incorrects, handler non chargé, barre rerender, CSS conflict).
2) Refactor pour avoir :
   - UI statique de la barre (HTML fixe)
   - JS handler robuste (dans /www/progress.js)
   - un `shinyjs::useShinyjs()` si besoin pour garantir la dispo JS
3) Valider que chaque clic (réponse) fait avancer le %.
